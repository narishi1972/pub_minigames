---

### 1-2. コードの品質

#### バグの可能性

1. **`showOverlay` の `startBtn` 参照問題（軽微）**
   - `showOverlay()` 内で `overlay.innerHTML` を書き換えた後、`document.getElementById('startBtn')` で新しいボタンを取得している。
   - ゲーム開始時に最初の `startBtn` に登録したイベントリスナー（`startBtn.addEventListener('click', startGame)` 行631）は DOM 書き換え後に参照を失うが、再度 `addEventListener` しているため問題は起きない。
   - ただし、複数回ゲームオーバーになるたびに `showOverlay` が呼ばれると、同一 `#startBtn` IDの新要素に毎回イベントリスナーが追加されるが、各ゲームオーバーで DOM ごと作り直しているため実質的な重複漏洩は発生しない。問題は軽微。

2. **`clearLines` のインデックス操作**
   - `r++` で同一行を再検査しているが、`board.splice(r, 1)` と `board.unshift(...)` を繰り返す方式は正しく動作する。ただし、`cleared` が4以上になることは理論上ないため `LINE_POINTS[4]` の配列境界も問題なし。

3. **`lockPiece` のゲームオーバー判定**
   - `current = next` に代入した後 `isValid(current, 0, 0, 0)` で衝突チェックしており、正常動作している。

4. **ソフトドロップ（下キー / S / K）未実装**
   - 標準的なテトリスに存在する「ゆっくり落下させる」操作が存在しない。プレイアビリティに影響する可能性がある。

#### 可読性

- IIFE（即時実行関数）による変数スコープ封じ込めは良好なプラクティス。
- 定数は先頭にまとめられており読みやすい。
- `PIECES` の回転定義が配列リテラルで明快。
- `isValid` の引数に差分（`dr, dc, dRot`）を渡す設計は汎用性が高く適切。
- 関数名が意図を明確に示している（`lockPiece`, `clearLines`, `hardDrop` 等）。

---

### 1-3. 改善提案

1. **ソフトドロップの追加**
   ```javascript
   case 'K': // または ArrowDown
     if (isValid(current, 1, 0, 0)) {
       current.row++;
       resetDropTimer();
       render();
     }
     break;
   ```

2. **`dropInterval` 変更後のタイマーリセット**
   - レベルアップ時に `dropInterval` が更新されるが、現在のタイマーはそのまま走り続ける。`clearLines()` 内でレベルが上がった際に `resetDropTimer()` を呼ぶべき。
   ```javascript
   if (cleared > 0) {
     // ... 既存処理 ...
     resetDropTimer(); // ← 追加
   }
   ```

3. **キーガイドとキーバインドの同期**
   - ゲーム画面左下のキーガイドに表示されているキーと実際のキーハンドラーは一致しているが、矢印キーも一般ユーザーには直感的なため、矢印キーのサポートを追加すると UX が向上する。

4. **ポーズ機能の追加**
   - Escキーなどでポーズできるとより完成度が高まる。

5. **スコアのハイスコア保持**
   - `localStorage` にハイスコアを保存することで、セッションを超えた記録保持が可能になる。

---

### 2-2. コードの品質

#### バグの可能性

1. **当たり判定の `getDinoArt('dead')` 呼び出し（軽微なロジックの不整合）**
   ```javascript
   const dinoArt = getDinoArt('dead'); // checkCollision() 内
   ```
   - 衝突判定時に常に `DINO_DEAD` アートを使ってピクセルレベルの衝突チェックをしている。実際の描画状態（走行中・ジャンプ中）とは異なるアートを使うため、判定精度が若干ずれる可能性がある。判定結果全体の大勢には影響しにくいが、厳密には現在の `dinoState` に基づいたアートを使うべき。

2. **水平移動の速度が `dt` 非依存（フレームレート依存）**
   ```javascript
   if (keys['J']) {
     dinoCol = Math.max(DINO_MIN_COL, dinoCol - 1);
   }
   ```
   - J/L の移動量は毎フレーム固定値 `1` であるため、フレームレートが高い環境では移動が速くなる。`dt` を掛けた移動量で補正するか、移動クールタイムを設けるべき。

3. **`obstacleSpawnCountdown` のフレームベース/時間ベースの混在**
   ```javascript
   obstacleSpawnCountdown = randomSpawnInterval() / speed; // 時間ベース
   obstacleSpawnCountdown -= dt;                           // フレーム差分
   ```
   - `randomSpawnInterval()` はフレーム数（30〜80）を返すが、`dt` は60fps基準の正規化値のため、実質的には秒数に近い単位になっており若干混乱しやすい。コメントで単位を明記すると良い。

4. **`COLS` の計算精度**
   ```javascript
   const COLS = Math.floor(window.innerWidth / 9.6);
   ```
   - 等幅フォントの文字幅を `9.6px` と仮定しているが、OS・ブラウザ・フォントによって実際の文字幅は異なる。画面端で文字が欠けたり、余白が生じたりする可能性がある。`canvas` + `measureText` での正確な文字幅取得が理想的だが、実用上は許容範囲。

5. **`hiScore` の localStorage 未保存**
   - ページリロードでハイスコアがリセットされる。

#### 可読性

- グローバルスコープに変数・関数が直接配置されている。IIFE やモジュール化で名前空間汚染を防ぐと良い。
- `// ========== セクション名 ==========` の区切りコメントが豊富で、大まかな構造は読みやすい。
- `gameState` の状態機械設計（`idle | playing | gameover`）は明確で追いやすい。
- `DINO_BODY`, `DINO_RUN1`, `DINO_RUN2`, `DINO_JUMP`, `DINO_DEAD` のアスキーアート定義が分かりやすい。
- `getDinoArt(state)` による状態と描画の分離は良いアプローチ。

---

### 2-3. 改善提案

1. **当たり判定の修正（現在のアートを使用）**
   ```javascript
   // checkCollision() 内
   const dinoArt = getDinoArt(dinoOnGround ? 'run' : 'jump'); // dead ではなく実際の状態
   ```

2. **水平移動のフレームレート非依存化**
   ```javascript
   const DINO_MOVE_SPEED = 8; // 列/秒
   if (keys['J']) {
     dinoCol = Math.max(DINO_MIN_COL, dinoCol - DINO_MOVE_SPEED * dt * (1/60));
   }
   ```
   ただし、列は整数として管理しているため、サブセル精度の浮動小数点位置も別途管理する設計変更が必要になる。

3. **IIFE によるグローバルスコープ保護**
   ```javascript
   (function() {
     // すべてのコードをここに格納
   })();
   ```

4. **`localStorage` によるハイスコード永続化**
   ```javascript
   // ゲーム開始時
   hiScore = parseInt(localStorage.getItem('dinoHiScore') || '0', 10);
   // 更新時
   localStorage.setItem('dinoHiScore', hiScore);
   ```

5. **`COLS` のリサイズ対応**
   - 現在 `COLS` はページロード時に1度だけ計算される。ウィンドウリサイズ時に再計算し、ゲームを再初期化する処理を追加すると健全性が増す。

6. **スポーン間隔の単位統一**
   - `randomSpawnInterval()` の戻り値の単位をコメントで明示するか、関数名を `randomSpawnIntervalFrames()` などに変更して混乱を防ぐ。

## 全体所感

| 項目 | Tetris | Dino |
|------|--------|------|
| 仕様充足度 | ★★★★☆ | ★★★★★ |
| コード品質 | ★★★★☆ | ★★★☆☆ |
| 可読性 | ★★★★★ | ★★★★☆ |
| 改善の余地 | 小（ソフトドロップ・タイマー同期） | 中（グローバル変数・衝突判定） |

両ゲームとも動作する完成度を持ち、日本語UIとキー操作説明も整備されている。
Tetris は IIFE によるスコープ管理や関数設計が優秀。Dino はグローバル変数の多用と当たり判定の軽微な不整合が改善余地として挙げられる。
いずれも単一 HTML ファイルとして即座にブラウザで動作する点は優れている。
